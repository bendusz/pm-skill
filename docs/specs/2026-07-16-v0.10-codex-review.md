# v0.10 ‚Äî `/pm-skill:codex-review` + `/pm-skill:codex-help` design

Date: 2026-07-16
Status: approved

## Goal

Add a `/pm-skill:codex-review` command that, when the OpenAI Codex CLI is installed and
authenticated, spawns one or more non-interactive Codex agents to review the codebase and writes
the results to timestamped markdown reports in an untracked folder. A second, independent model's
review complements the plugin's Claude-based review panel.

## Research grounding (verified against codex-cli 0.145.0-alpha.4, 2026-07-16)

- `codex review` (alias of `codex exec review`) is the native non-interactive review primitive.
  It is read-only by design ("reports prioritized findings without modifying your working tree").
- Scope flags are **mutually exclusive with each other and with a custom PROMPT** (clap
  `conflicts_with_all`): `--uncommitted` | `--base <branch>` | `--commit <sha>` | `PROMPT`.
  Therefore objective-focused reviews must use the *prompt-as-scope* form: the prompt states both
  the diff scope and the focus (e.g. "Review the uncommitted changes. Focus exclusively on ‚Ä¶").
- `codex review` accepts `-m/--model`, `-c key=value`, `--json`, `-o/--output-last-message <file>`,
  `--ephemeral`, `--skip-git-repo-check`; it does **not** accept `-C/--cd` or `--sandbox` ‚Äî run it
  from the repo root.
- Reasoning effort has **no dedicated flag**: use `-c model_reasoning_effort="<level>"`.
  Levels: `minimal|low|medium|high|xhigh` (+ `max`/`ultra` on the GPT-5.6 generation,
  model-dependent).
- Whole-codebase review is not native ‚Äî use `codex exec --sandbox read-only` (exec's default,
  pinned explicitly) with a structured audit prompt.
- Preflight: `codex login status` exits non-zero when logged out; an unauthenticated `codex exec`
  burns ~20 s in retries before failing ‚Äî gate on login status first.
- Exit codes: 0 success, 1 runtime/auth failure, 2 CLI usage error. Progress goes to stderr; the
  final message to stdout / `-o` file.
- Current models: `gpt-5.6-sol` (flagship), `gpt-5.6-terra` (balanced), `gpt-5.6-luna` (fast),
  plus `gpt-5.4`, `gpt-5.4-mini`, `gpt-5.3-codex`. No dedicated review model; `review_model`
  config key exists but this command never touches the user's `~/.codex/config.toml`.

## Decisions (user-approved)

| Decision | Choice |
|---|---|
| Default model / effort | `gpt-5.6-terra` at `high` |
| `recent` scope meaning | last commit (`codex review --commit HEAD`) |
| Multi-objective report shape | one file per objective + a small index file |
| Objective specification | free-form text **and** curated presets |

## Command interface

```
/pm-skill:codex-review [scope] [model=...] [effort=...] [timeout=...] [objectives‚Ä¶]
```

- **scope** (default `worktree`):
  - `recent` ‚Üí `codex review --commit HEAD`
  - `worktree` ‚Üí `codex review --uncommitted`
  - `codebase` ‚Üí `codex exec --sandbox read-only` + whole-repo audit prompt
- **model=** model id passed via `-m` (default `gpt-5.6-terra`).
- **effort=** passed via `-c model_reasoning_effort="‚Ä¶"` (default `high`).
- **timeout=** per-agent timeout in minutes (default 10).
- **objectives**: zero or more of the presets `security`, `bugs`, `architecture`, `tests`,
  `performance`; `panel` expands to all five; any other quoted/free text becomes a custom
  objective. Zero objectives ‚Üí one general review using the native scope flags.

Each objective spawns one Codex process, all in parallel (background Bash). Multi-objective runs
use the prompt-as-scope form; a no-objective run uses the native scope flag.

## Execution flow

1. **Preflight** ‚Äî `command -v codex`; `codex login status` (exit-code gated); git repo check.
   Missing codex / logged out ‚Üí stop with install (`npm i -g @openai/codex` or
   `brew install codex`) / `codex login` instructions. Diff scopes (`recent`, `worktree`) require
   a git repo; `codebase` works outside one with `--skip-git-repo-check`.
2. **Output dir** ‚Äî use `untracked/` if it exists at repo root; else create `codex/` and, if the
   repo is tracked, ensure `codex/` is in `.gitignore` (append when absent).
3. **Spawn** ‚Äî from the repo root, one background process per objective:
   `codex [exec] review ‚Ä¶ -m <model> -c model_reasoning_effort="<effort>" --ephemeral
   -o <outdir>/<stamp>-codex-review-<scope>[-<objective>].md` (no `--color`/`--sandbox`/`-C` ‚Äî
   `exec review` rejects them), stderr ‚Üí sidecar
   `‚Ä¶.stderr.log`. Stamp format `YYYY-MM-DD-HHMM`. Monitor until all exit or time out.
4. **Report** ‚Äî
   - single review: the one `-o` file is the report;
   - multi-objective: per-objective files plus `<stamp>-codex-review-<scope>-index.md` with run
     metadata (model, effort, scope, per-objective exit code and a one-line finding summary).
   Failures are recorded in the index; partial results are kept ‚Äî one failed objective never
   discards the others. Sidecar stderr logs are deleted on success, kept on failure.
5. **Chat summary** ‚Äî report file paths plus the top findings per objective.

## Preset objective focuses

- `security` ‚Äî authn/authz gaps, injection, secret handling, unsafe deserialization, dependency
  risk.
- `bugs` ‚Äî correctness: logic errors, edge cases, error handling, race conditions, silent
  failures.
- `architecture` ‚Äî module boundaries, coupling, abstraction fit, structural drift.
- `tests` ‚Äî coverage of changed behavior, missing edge cases, assertion quality, flakiness risk.
- `performance` ‚Äî algorithmic complexity, N+1 patterns, unnecessary allocation/IO, hot paths.

## Second command: `/pm-skill:codex-help`

Ask Codex a specific question ‚Äî advice, a recommendation, a second opinion, help with a decision ‚Äî
about the current work. Claude composes the question with the relevant context (files, tradeoffs,
constraints) and relays Codex's answer in chat; no report file.

```
/pm-skill:codex-help [model=...] [effort=...] <question>
```

- Default model `gpt-5.6-sol` at `medium` effort (a *higher-tier* default than codex-review ‚Äî this
  is judgment work), both overridable.
- Mechanics: `codex exec --sandbox read-only --ephemeral -o <scratch file>` from the repo root so
  Codex can read the codebase while answering; same preflight as codex-review (binary, login;
  `--skip-git-repo-check` outside a repo).
- **Usage guardrail (user requirement):** not for routine questions ‚Äî reserve it for real,
  consequential changes that warrant a second pair of eyes (design decisions, risky refactors,
  tricky tradeoffs). The command text and the SKILL.md note both say so.
- If `pm/log.md` exists, append a one-line entry (question gist + one-line answer gist).

## Out of scope

- No changes to the delivery loop's mandatory review panel ‚Äî the project-manager skill gains only
  a one-line note that `/pm-skill:codex-review` is available as an optional independent
  second-model reviewer.
- No `review_model`/config.toml management, no auth flows beyond detection, no JSON parsing of
  the `--json` event stream (the `-o` final message is the report).
