# pm-skill v0.4 — Delivery agents (Tier-2) + optional model tiering

- **Status:** Approved (scope) — 2026-06-04 · builds on v0.3.0
- **Type:** increment to the public `pm-skill` plugin

---

## 1. Goal
Round out the delivery roster with three more specialist agents and give users an optional,
**generic** way to control model cost — without changing any default behaviour. Everything here is
additive: a bare install with none of these triggered behaves exactly as v0.3.0 did.

## 2. Scope
- **Three new agents:** `security-auditor`, `technical-writer`, `debugger`.
- **Wiring:** promote the security lens in `review-gates.md`; add `debugger` to the fix step and an
  optional Document step in `implementation-loop.md`; list the new agents in `SKILL.md`.
- **Optional model tiering:** a new `references/model-tiering.md` (abstract tiers, no vendor model
  IDs); agents stay `model: inherit`.
- **Completion report:** `templates/completion-report.md.template`, produced by `technical-writer`.
- Version → `0.4.0`.
- **Deferred (out of scope):** EARS/INVEST discovery rigor → later increment; parallel `[P]` story
  execution via git worktrees → `v0.5.0` (the risky one).

## 3. The three agents (same conventions as the existing five)
All use `model: inherit`, declare `name`+`description` (with an `<example>`), return a tight
structured summary, and carry no third-party names (keeps the artifact generic).

### 3.1 `security-auditor` — read-only deep security lens
- **Tools:** `Read, Grep, Glob` (read-only). **Color:** orange.
- A *deeper* pass than `code-integrity-reviewer`'s baseline security check. **Risk-triggered**, joins
  the review panel only for stories touching auth/authz, crypto, secrets/credentials, external or
  untrusted input, file/network/process I/O, deserialization, or dependency changes.
- Same I/O contract as the other reviewers: gets the story file + the diff **text** the PM generated
  (it has no Bash to diff), returns `block`/`major`/`minor` findings + a `PASS`/`CONCERNS`/`FAIL`
  verdict, so it aggregates into the existing panel unchanged.
- Focus: concrete, exploitable issues (injection, authz gaps, secret handling, unsafe
  deserialization, path traversal, SSRF, weak crypto, vulnerable/oudated deps) — not generic advice.

### 3.2 `technical-writer` — docs only, never source
- **Tools:** `Read, Write, Edit` (docs only — mirrors `test-engineer`'s "tests only" discipline).
  **Color:** yellow.
- Writes/updates **user-facing docs** from shipped work: README sections, usage/how-to docs,
  CHANGELOG entries, and the project **completion report**. Touches only docs (`README*`, `docs/`,
  `CHANGELOG*`, `*.md` docs) — never source or tests.
- Runs **opt-in at a sprint or project boundary**, not per story. Inputs: the plan, `tmp/log.md`, and
  the relevant story files. Returns: files written + a one-line summary of each.

### 3.3 `debugger` — read-only root-cause diagnostician
- **Tools:** `Read, Grep, Glob` (read-only — **no Bash**, **no edits**). **Color:** pink.
- Invoked when a deterministic gate fails or the fix loop stalls. The PM (who already ran the gate)
  hands it the **failing command's output**, the **diff text**, and the relevant paths. It returns a
  **root-cause diagnosis + a concrete fix plan** (`file:line`, the cause, the minimal suggested
  change). It does **not** patch — `expert-builder` still owns the fix.
- Rationale (Decision A): keeping it read-only preserves "the builder is the single writer" and the
  "PM runs commands, subagents reason" model, and stays consistent with the reviewers (no Bash). It
  breaks the fix loop's "blind retry" failure mode by inserting a diagnosis step within the cap.

## 4. Wiring into the existing flow (edits — inert unless triggered)
- **`review-gates.md`:** promote `security-auditor` from the "extensible" parenthetical to a
  first-class **risk-triggered lens** with explicit triggers (above). Aggregation rule is unchanged
  (story passes only when every selected lens has no open `block`/`major`). Note that
  `technical-writer`/`debugger` are *delivery* agents, not review lenses.
- **`implementation-loop.md`:**
  - **Step 4 (Fix):** if a gate fails or the loop doesn't converge after a round, dispatch `debugger`
    to root-cause **before** the next builder attempt — still inside the existing **3-round** cap.
  - **New optional Step 8 (Document):** at the sprint/project boundary, optionally dispatch
    `technical-writer` to update docs and produce the completion report. Optional and logged.
- **`SKILL.md`:** add the three agents to "Agents you orchestrate"; add a one-line pointer to
  `references/model-tiering.md`.

## 5. Optional model tiering (generic, opt-in) — `references/model-tiering.md`
- **Default unchanged:** every agent stays `model: inherit`; out of the box nothing changes.
- Provides an **abstract tier table — no vendor model IDs** — so it stays publishable and
  machine-agnostic:
  - **deep** — planning, architecture review, security audit, debugging root-cause.
  - **standard** — build, code-integrity review, tests.
  - **light** — logging, formatting, doc boilerplate.
- Explains *how* a user opts in (map tiers → their own models via Claude Code's per-agent `model`
  override / settings), and *why* the bundled agents ship as `inherit` (generic + zero surprise).
  Guidance only — the skill never auto-selects models.

## 6. Completion report — `templates/completion-report.md.template`
Sections: overview; what shipped (per sprint/story); acceptance-criteria status; gates & review
summary; follow-ups / known issues; links (plan, log, PRs/merges). Produced by `technical-writer` at
project end, or filled by the PM directly. Gitignored? No — it's a deliverable, lives under `docs/`.

- Merge `--no-ff` to `main`; tag `v0.4.0`; push; `gh release create`.

## 8. Decisions log
- **A — `debugger` is read-only** (no Bash, no edits): diagnoses and hands a fix plan to
  `expert-builder`; the builder stays the single writer.
- **B — model tiering is optional guidance with abstract tiers**, no shipped defaults and no vendor
  model IDs: keeps the plugin generic and behaviour unchanged by default.
- **C — the completion report ships in v0.4.0**, paired with `technical-writer` as its flagship
  output.
- `security-auditor` reuses the existing reviewer I/O contract and panel aggregation — no new
  machinery, just another risk-selected lens.
