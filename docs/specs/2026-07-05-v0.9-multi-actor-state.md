# v0.9 — Multi-actor PM state (design)

**Date:** 2026-07-05 · **Status:** approved (Ben, 2026-07-05) · **Target:** pm-skill 0.9.0

## Problem

Since v0.8 the PM state trio (`pm/pm-state.json`, `pm/log.md`, `pm/HANDOFF.md`) is git-tracked, which
made it durable — and introduced a collaboration failure: the files are **single-actor by design**
(one `current_story`, one `next`, one overwrite-only handoff, one log whose Current State block both
people would edit). Two developers each running PM sessions on the same repo overwrite each other's
state and conflict on every log append.

## Decisions (agreed 2026-07-05)

1. **Target model:** multiple people run concurrent PM sessions against one shared plan.
2. **Log:** one shared `pm/log.md`, entries author-prefixed, concurrent appends auto-merged via a
   `merge=union` gitattribute.
3. **State:** shared project core + per-actor position files.
4. **Layout:** one unified layout — solo is a team of one. No solo/team mode switch.
5. **Coordination level ("Approach B"):** remove mechanical friction (file split, union merge, sync
   discipline) and *surface* human coordination (analyze/doctor/session-context visibility, one
   accident tripwire). No hard claim locking — git has no locks; a same-minute double-claim race
   stays possible but becomes visible within one pull.

## File layout

```
pm/
  pm-state.json            # SHARED project state (slimmed — see schema)
  log.md                   # SHARED logbook: header + append-only author-prefixed entries
  actors/
    <actor-id>.json        # per-person working position
    <actor-id>.HANDOFF.md  # per-person handoff briefing (overwrite per handoff, as today)
.gitattributes             # gains: pm/log.md merge=union   (bootstrap; append, don't clobber)
tmp/                       # unchanged: gitignored scratch, never load-bearing
```

### Actor identity

Derived, never configured: slug of git `user.email` local part; fallback slug of `user.name`;
fallback `unknown-actor` (with a doctor warning). Slug = lowercase, non-alphanumerics → `-`,
e.g. `v.bende@gmail.com` → `v-bende`. Every command/hook that needs "who am I" derives it the
same way.

## Schemas

### `pm/pm-state.json` (shared — changes only at coordination moments)

```json
{
  "project": "<name>",
  "spec": null,
  "constitution": null,
  "scale": "standard",
  "phase": "discovery",
  "signed_off": false,
  "approver": null,
  "approved_date": null,
  "integration_branch": "main",
  "current_sprint": null,
  "total_sprints": null,
  "last_analysis_status": null,
  "assignments": {},
  "updated": "<YYYY-MM-DD HH:MM>"
}
```

- `assignments` maps story id → actor id for **active claims only**; the entry is removed in the
  same commit that ships the story. History lives in the log and the story files.
- Removed from the shared file (moved to actor files): `current_story`,
  `current_story_status`, `current_story_verification_status`, `current_story_rounds`,
  `current_story_retries`, `branch`, `parallel_batch`, `next`, `handoff_written`.

### `pm/actors/<actor-id>.json` (per person)

```json
{
  "actor": "<actor-id>",
  "current_story": null,
  "current_story_status": null,
  "current_story_verification_status": null,
  "current_story_rounds": null,
  "current_story_retries": null,
  "branch": null,
  "parallel_batch": [],
  "next": "<continuation point>",
  "handoff_written": null,
  "updated": "<YYYY-MM-DD HH:MM>"
}
```

Field semantics are unchanged from v0.8 (bound counters persist across sessions; `handoff_written`
vs `updated` is the handoff staleness check — both now scoped to this actor).

### `pm/log.md` (shared, append-only)

```
# PM Log — <project>

- <YYYY-MM-DD HH:MM> <actor-id> — <2–3 sentence entry, written for a colleague with zero context>
```

**The Current State block is removed.** It was per-person and mutable — the two properties that
made concurrent editing conflict. "Where things stand" is rendered at resume/session-start from the
state JSONs instead. Union merge may occasionally reorder near-simultaneous entries; harmless for
an append-only log, and exactly why no mutable block may live in this file.

## Claim & sync discipline (behavioral — checked, not enforced)

- **Pull/rebase before claiming** a story and **before shipping** one.
- **Claim** = write `assignments[story] = you` + create the story branch, commit the claim promptly
  and push it (when a remote exists), so teammates see the claim within one fetch.
- If the integration tip moved after your gates ran, **re-gate on the merged result before merging**
  (the parallel path's semantic-conflict rule, applied across people).
- **Release** the claim in the ship commit (assignment removed, log entry appended).
- **Sprint advance:** any actor may advance `current_sprint` once all the sprint's stories are
  merged; it is a shared-state write and gets a log entry. Checkpoint policy (sprint/story/autonomous
  pauses) is unchanged.
- Sign-off remains global and one-time; `/pm-skill:correct-course` voiding it is a shared-state
  write that affects every actor (the sign-off hook re-engages for all).

## Hooks

| Hook | Change |
| --- | --- |
| `require-signoff.sh` | None in behavior — still reads shared `pm/pm-state.json` `signed_off` (plus existing legacy `tmp/` fallback). `pm/*` allowlist already covers `pm/actors/`. |
| `pm-secrets-guard.sh` | None — already guards all of `pm/`, which now includes actor files. |
| `session-context.sh` | Extend: derive actor id; print shared position (phase, sprint, sign-off), **your** actor position and handoff freshness, and teammates as read-only one-liners (`anna: S2-3 in review · pm/S2-3-…`). Legacy flat/tmp layouts keep their current messages. |
| `actor-guard.sh` (**new**) | Fail-open tripwire: block Write/Edit to `pm/actors/<id>.json` / `<id>.HANDOFF.md` when `<id>` ≠ the current actor id. Same `PM_SKILL_NO_ENFORCE=1` kill switch, jq-optional, silent outside `pm/actors/`. Shared files (`pm-state.json`, `log.md`) stay writable by everyone. |

Story-claim conflicts are **surfaced, not blocked**: hard locking would false-block legitimate work
(pairing, takeover after illness, stale claims) and git cannot make it atomic anyway.

## Command / reference touch points

- **`resume`** — read shared state + your actor file + your handoff (staleness check unchanged,
  actor-scoped); render teammates' positions; migration step (below) runs here first when needed.
- **`handoff`** — writes `pm/actors/<you>.HANDOFF.md`, sets `handoff_written` in your actor file.
- **`doctor`** — team health: double-claims, claims with no matching branch (stale), actor files
  matched by `.gitignore`, missing `merge=union` rule, unparseable actor JSON, actor id derivable.
- **`analyze`** — new checks: two actors claiming one story; cross-actor in-flight stories with
  overlapping `Touches`; assignment pointing at a nonexistent story or actor file.
- **`correct-course`** — unchanged flow; note that voiding sign-off halts all actors.
- **`logging-and-state.md`** — rewrite around shared-core + actors (layout, schemas, log format,
  claim/sync discipline, migration).
- **`planning-and-signoff.md`** — bootstrap: create `pm/actors/`, write the `.gitattributes` union
  rule (append; don't clobber), same check-ignore file checks extended to the actor files.
- **`implementation-loop.md`** — step 0 claims the story (pull → assign → branch → push claim);
  step 7/8 releases the claim in the ship commit; counters live in your actor file.
- **`parallel-execution.md`** — `parallel_batch` moves to the actor file; multi-actor note: the
  batch is per-actor, cross-actor coordination is via `assignments`.
- **Templates** — slim `pm-state.json.template`; new `actor-state.json.template`; `log.md.template`
  loses the Current State block; `HANDOFF.md.template` and `CLAUDE.md.template` path/wording updates.
- **Example** — `examples/todo-cli/pm/` refreshed to team-of-one (`actors/bendusz.json` + handoff);
  ideally show a second actor entry in `assignments`/log to illustrate the team shape.
- **README / CHANGELOG / plugin.json** — team story, 0.9.0.

## Migration (0.8 → 0.9)

Trigger: any state read (resume, session hook prints a pointer) where `pm/pm-state.json` contains
personal fields (`current_story` et al.) and `pm/actors/` does not exist.

1. Derive your actor id; create `pm/actors/<you>.json` from the personal fields; slim the shared file.
2. Seed `assignments` from `current_story` (if one is in flight).
3. Move `pm/HANDOFF.md` → `pm/actors/<you>.HANDOFF.md` (if present).
4. Strip the log's Current State block (its live content is now in the state files); keep all
   existing log entries verbatim — only new entries carry the actor prefix.
5. Append the `merge=union` rule to `.gitattributes`.
6. Log the migration and commit.

The pre-0.8 `tmp/` migration chains in front (tmp → flat pm → actors) so any vintage resumes in one
pass. `require-signoff.sh` needs no flat-layout fallback beyond what it has: `signed_off` stays in
the shared file at the same path.

## Known limitations (accepted)

- **No atomic claims.** Two actors claiming one story between pulls race; detection is one fetch
  away (claim commits + doctor/analyze), resolution is human.
- **Union merge ordering.** Near-simultaneous log entries can interleave out of timestamp order.
- **Actor id drift.** Changing git identity mid-project creates a second actor file; doctor flags
  actor files with no matching recent git author.

## Testing

- Functional hook tests (scratch repos, as in 0.8): actor-guard allows own files / blocks others' /
  silent elsewhere; session hook renders self + teammates; sign-off and secrets guards regression.
- Scripted migration dry-run: 0.8 flat layout → 0.9 (fields split, log block stripped, gitattributes
  appended); and chained tmp → 0.9.
- Two-actor simulation: two clones, concurrent claims + log appends, verify union merge and
  doctor/analyze detection of a manufactured double-claim.
- `scripts/validate.sh` + shellcheck stay green; validator gains a check that the new template
  parses.
