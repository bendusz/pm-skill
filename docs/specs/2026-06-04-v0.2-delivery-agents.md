# pm-skill v0.2 — Delivery Agents & Risk-Based Review

- **Status:** Approved (scope) — 2026-06-04 · builds on v0.1.0
- **Type:** increment to the public `pm-skill` plugin

---

## 1. Goal
Add **role-based** delivery agents and turn single-reviewer review into a **risk-selected panel**,
so the PM can match the right specialists to each story without running every lens on every change.

## 2. Scope
**In (v0.2):**
- New bundled agents: `codebase-analyst`, `test-engineer`, `architecture-reviewer`.
- `review-gates.md` gains **reviewer selection by risk** (a panel, not a single reviewer).
- `planning-and-signoff.md` gains an optional **analyze existing code** step (brownfield).
- `implementation-loop.md` notes the optional `test-engineer` step and the review **panel**.
- Version → `0.2.0`; CHANGELOG entry.

**Deferred (next increment, Tier-2):** `security-auditor`, `technical-writer`, `debugger`.
**Out (niche/domain):** performance-reviewer, dependency auditor, accessibility/UX, DB-migration.

## 3. Principles (unchanged from v0.1)
- **Role-based, not domain-based.** `expert-builder` still *adopts* the needed specialty; we add
  distinct delivery **roles**, never a zoo of domain experts.
- **Reviewers are read-only** (`Read, Grep, Glob`); the PM still runs the deterministic gates itself.
- **Protect context:** each agent gets only what it needs and returns a structured summary.
- **Risk-selected panel:** don't run every lens on every story.

## 4. The new agents

### 4.1 `codebase-analyst` (read-only) — `Read, Grep, Glob`, model `inherit`, color `cyan`
- **Role:** map an existing/brownfield codebase into a **context pack** the PM can embed into the
  plan and into self-contained story files.
- **Returns:** architecture & module boundaries; conventions (naming, error handling, patterns);
  the real **test/lint/build/run commands** (read from config files); where new code should live;
  existing patterns to follow; risks/landmines. Structured, concise — not a file dump.
- **Used in:** planning (brownfield) and per-story when a story touches unfamiliar code.

### 4.2 `test-engineer` (writes **tests only**) — `Read, Write, Edit, Bash, Grep, Glob`, model `inherit`, color `green`
- **Role:** author tests from a story's **acceptance criteria**, independent of the implementer
  (removes the "tests encode the same misunderstanding as the code" blind spot).
- **Behavior:** writes ONLY tests, never implementation; may run **before** build (TDD red) or
  **after** (coverage/characterization). Runs the tests to report their state.
- **Returns:** tests added (paths), how to run them, current pass/fail, gaps it could not cover.

### 4.3 `architecture-reviewer` (read-only) — `Read, Grep, Glob`, model `inherit`, color `purple`
- **Role:** a higher-altitude lens than `code-integrity-reviewer`: module boundaries, abstractions,
  cohesion/coupling, over-engineering / YAGNI, tech-debt drift, and fit to the intended architecture.
- **Returns:** the same shape as the integrity reviewer — severity-graded findings
  (`block`/`major`/`minor`) + a `PASS`/`CONCERNS`/`FAIL` verdict.

## 5. Workflow integration

**Planning** — optionally dispatch `codebase-analyst` first on an existing codebase; fold its
context pack into `docs/plan.md` (architecture/commands) and later into story files.

**Implementation loop** — optional `test-engineer` step: before build for acceptance tests (TDD),
or after build to harden coverage. The **Review** step becomes a panel (below).

**Review gates — selection by risk.** The PM chooses lenses per story:
- **Always:** `code-integrity-reviewer` (correctness + security baseline).
- **Add `architecture-reviewer`** when the story changes structure: new modules, refactors,
  cross-cutting changes, new abstractions/interfaces. **Skip** it for trivial/localized changes.
- Each lens is a **separate agent** (multi-lens verification); the PM aggregates verdicts. A story
  passes review only if every selected lens is `PASS`/`CONCERNS` with no open `block`/`major`.
- The framework is extensible: when Tier-2 lands, `security-auditor` joins for
  auth/crypto/external-input/secret/dependency stories, and a perf lens for hot paths.

The PM still runs the deterministic gates (test/lint/build) itself, per the v0.1 hard rule.

## 6. Quality & release
- `claude plugin validate`; generic-clean check (no third-party plugin names under `plugins/pm-skill/`).
- Subagent **dry-run** of the three new agents and the changed reference files; fix findings.
- Merge to `main`; tag `v0.2.0`.

## 7. Decisions log
- v0.2 ships the **three Tier-1** agents + **risk-based review panel**; Tier-2 deferred.
- New reviewers/analyst are **read-only**; `test-engineer` writes **tests only**.
- Review generalised from a single reviewer to a **risk-selected, multi-lens panel**.
