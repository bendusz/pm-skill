# pm-skill v0.5 — Parallel `[P]` story execution via git worktrees

- **Status:** Approved (scope + research-backed decisions) — 2026-06-04 · builds on v0.4.0
- **Type:** increment to the public `pm-skill` plugin

---

## 1. Goal
Activate the `[P]` (parallel-safe) marker that decomposition already records. Let the PM build
several independent stories at once in isolated git worktrees, then integrate them **one at a time**
through the existing gate/review/fix machinery — without weakening any of the 8 hard rules. Opt-in
and best-effort: if anything is unavailable or goes wrong, fall back to today's sequential loop.

## 2. Why this is safe (the load-bearing insight)
Builders **make no commits** (hard rule 1/8: the PM owns git). So the concurrent phase does **zero
git writes** — N builders only edit files in N separate worktree directories. Every git operation
(worktree add/remove, commit, merge) is **PM-owned and serial**. This sidesteps the documented
failure mode of parallel agents (shared `.git` lock contention when many processes commit at once —
e.g. Claude Code issue #55724: 13 concurrent committers → 8 failed). Here the concurrent phase is
pure file-editing; integration is serial.

## 3. Scope
- **New reference:** `references/parallel-execution.md` — the full protocol.
- **Edits:** `implementation-loop.md` (sprint-start branch point: parallel vs sequential),
  `decomposition.md` (`[P]` now activates parallel exec + an optional **Touches** hint),
  `logging-and-state.md` + `pm-state.json.template` (a `parallel_batch` array), `SKILL.md` (pointer +
  one safety line), `story.md.template` (optional **Touches** field).
- Version → `0.5.0`; CHANGELOG.
- **Deferred / out of scope:** per-worktree runtime isolation (ephemeral DB/port-offset harnesses);
  agents that self-merge or open their own PRs; concurrent *authoritative* test runs. EARS/INVEST
  discovery rigor remains a later increment.

## 4. Decisions (researched — industry + community)
- **A — Concurrent build, serialized integration; the serial run is authoritative.** Matches every
  mainstream system: GitHub merge queue, GitLab merge trains, Bors / Graydon Hoare's "Not Rocket
  Science Rule", Google/Meta submit queues, and Anthropic's multi-agent guidance ("3–5 subagents;
  parallelism underperforms when tasks share context/dependencies"). Full-concurrency merging is the
  unsafe path the industry built tooling to avoid.
- **B — Default fan-out 3, configurable up to ~5.** Anthropic "3–5 subagents"; Claude Code Agent
  Teams "start with 3–5"; Dave Paola's measured degradation (2 occasional / 5 frequent / 9
  pathological conflicts; conflict surface ~N²). The 16-/hundreds-of-subagents caps are for
  read-only fan-out, not concurrent writers.
- **C — Pessimistic conflict net.** Worktrees isolate files, not logic (Fowler's *semantic
  conflict*: each branch green alone, broken together). **Before each land: merge the latest
  integration tip into the story branch and re-run the full gates; land only on green.** Order: FIFO
  / priority, with true dependency-order where a real dependency exists. Add an **up-front
  file-domain overlap check** — if two `[P]` stories would write the same files, serialize them.
  Optional `git rerere` to dedupe repeated resolutions. No source endorses optimistic merge for
  automated fan-out.
- **D — Worktrees in a gitignored in-repo dir: `tmp/worktrees/<branch-slug>/`** (already inside the
  skill's gitignored `tmp/`). **Prefer the host's native worktree isolation when available** (Claude
  Code ships `.claude/worktrees/` + `isolation: worktree`); document a sibling/external dir as an
  opt-out for IDEs that over-index. (Research split — classic "sibling" vs agent-tooling "in-repo
  gitignored" — resolves toward in-repo gitignored for a Claude Code plugin.)

## 5. The parallel path (built on A)
**Eligibility (checked at sprint start).** A parallel batch = stories in the sprint that are
**build-ready**, marked **`[P]`**, whose `depends-on` are **already merged**, and that **don't share
a file domain** (Touches overlap → serialize the colliding ones). Need ≥2 after that filter, and
`git worktree` must work. Otherwise run the existing sequential loop.

**Bounded fan-out.** Cap concurrent builders (**default 3**, configurable). Larger sets run in waves.

**Build phase (parallel, edit-only).** For each story: create a worktree + story branch from the
integration branch (`git worktree add tmp/worktrees/<slug> -b pm/S<n>-<k>-<slug> <integration>`).
Dispatch the batch's builders **together** (concurrent subagent calls — one orchestration step),
each given only its story file + its worktree path, told to **implement and self-check only, not run
the full/exclusive suite** (the PM runs authoritative gates serially next; this avoids concurrent
port/DB collisions on a generic project). Collect only structured summaries. Then **commit each
story's edits to its story branch** (story-path-scoped; never `git add -A`) so the work is captured
before any worktree is removed. If the host serializes the dispatch, isolation still holds — only the
wall-clock win is lost, not correctness.

**Integration tail (serial — FIFO/priority, dependency-order where real).** For each built story, in
order:
1. Bring the latest integration tip into the story branch (merge/rebase).
2. Run the project's **authoritative** gates on that combined result (test/lint/build per
   `review-gates.md`) — this is where a semantic/cross-story conflict surfaces.
3. Produce the story-scoped diff → run the **review panel** (risk-selected, incl. `security-auditor`
   when warranted) → **triage** → **fix** via the builder, **≤3 rounds** (dispatch `debugger` first
   when a gate fails or a round stalls), re-gating each round.
4. On green + no open `block`/`major`: `--no-ff` merge the story branch into the integration branch
   (local default; remote PR only on explicit request — unchanged). **Log.**
5. **Remove the worktree** (`git worktree remove`) now that the work is committed and merged.

A blocked story **escalates** per the normal cap without blocking the rest; a partial batch still
checkpoints at the sprint boundary (default checkpoint cadence unchanged).

## 6. Worktree lifecycle & safety invariants (bold in the reference)
- **Create** with `git worktree add -b`; never check the same branch out twice (give each its own).
- **Capture before cleanup:** commit a story's edits to its branch *before* removing its worktree.
- **Remove** only with `git worktree remove` — **never `rm -rf`**. **Never force-remove a worktree
  with uncommitted changes** (check `git -C <wt> status --porcelain` first; preserve + report) — this
  is the documented #1 data-loss path.
- **No orphans:** clean up every worktree the PM created, on success *or* error/interruption; sweep
  with `git worktree prune` on entry/resume.
- **Don't run concurrent `git gc`.** Keep `tmp/worktrees/` gitignored (ensure the entry exists
  before creating any worktree).
- Unchanged: clean-tree precondition, story-path-scoped diffs/commits, never push without explicit
  request, separate reviewer, bounded loops, ask-before-`git init`.
- **Sign-off hook:** worktrees exist only post-sign-off, and `tmp/pm-state.json` lives in the main
  checkout — so the hook fail-opens inside a worktree, which is correct (`signed_off` is already
  `true` by then). Noted explicitly so it isn't mistaken for a gap.

## 7. State
`pm-state.json` gains a `parallel_batch` array — each entry `{story, branch, worktree, status}` with
status `building | built | in-review | merged | blocked` — alongside the existing single-story
fields (sequential path). On resume, reconcile it against `git worktree list` and `tmp/log.md`.

## 8. Fallback & known limitations (documented, not silent)
- **Hard fallback:** parallel is used only when eligible *and* worktrees work; any failure → finish
  that story sequentially. Sequential remains the default and the safety net ("detect and adapt,
  never depend").
- **Runtime isolation:** worktrees don't isolate ports/DBs/caches. Mitigated generically by making
  the **authoritative** test run serial; true concurrent testing needs project-provided isolation
  (out of scope).
- **Worktree bootstrap:** a fresh worktree doesn't carry untracked files (`.env`, `node_modules`).
  Concurrent builders therefore do light self-checks only; if a worktree needs deps for even that,
  the PM bootstraps it, or the story runs sequentially.
- **Split areas flagged in the reference:** shared-`.git` severity (transient-lock vs corruption —
  we avoid it by edit-only builders), worktree location (in-repo vs sibling), self-merge vs central
  (we keep central/PM-owned). Conservative default chosen in each.

`gh release create`.

## 10. Decisions log
- **A** concurrent edit-only build + **serialized** integration; serial run authoritative.
- **B** default fan-out **3** (configurable ≤5).
- **C** pessimistic net: merge-latest-in + full re-gate before each land; file-domain overlap check;
  optional `rerere`; FIFO/priority/dependency order.
- **D** `tmp/worktrees/` (gitignored, in-repo); prefer native isolation when present; sibling as
  opt-out.
- The **edit-only builder** rule is what makes concurrent building safe — it is the design's
  keystone, not an incidental hard rule.
